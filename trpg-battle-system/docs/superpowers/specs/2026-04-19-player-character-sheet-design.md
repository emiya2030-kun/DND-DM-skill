# Player Character Sheet Under Battlemap Design

## 目标

在 localhost battlemap 页面中，为“固定主角”新增一块位于**地图下方**的角色卡区域。

第一版目标不是完整角色页，而是先做出一个可用、可扩展、视觉上接近 CRPG / 博德之门风格的主角卡，优先满足：

- 地图下方能稳定显示固定主角卡
- 左侧有主角头像位
- 右侧有 RPG 风格主面板
- 底部用按钮式页签切换内容，而不是一次性铺满所有表格
- 第一版先覆盖核心字段
- 第二版再切到动态数据

## 范围

### 本轮纳入

- 修改 localhost battlemap 页面布局
- 在地图下方新增主角卡 UI
- 第一版固定显示一个主角角色卡
- 角色卡使用统一 `player_sheet` 前端数据对象驱动
- 底部提供 3 个页签按钮：
  - `技能`
  - `装备`
  - `后续追加`
- 第一版先用静态数据驱动
- 页面状态刷新时角色卡同步刷新，但保留当前激活页签

### 本轮不纳入

- 多角色切换
- 点击地图单位切换角色卡
- 完整装备系统展示
- 完整特性、状态、资源、法术位面板
- 后端正式产出完整 `player_sheet` schema
- 上传/管理真实头像资源后台

## 用户确认的关键决定

- 角色卡显示对象：**固定主角**
- 第一版策略：**先静态做出 UI，再接动态数据**
- 视觉方向：**厚重金属战场感材质 + 更清晰的 RPG 分区排版**
- 信息范围：**先做核心字段，不上完整角色页**
- 交互结构：**按钮切换式页签**
- 顶部摘要中：`44 / 44 HP` 后直接显示 `AC 16`

## 当前代码上下文

当前 localhost battlemap 页面主要由以下位置生成：

- `scripts/run_battlemap_localhost.py`
  - 负责拼接 localhost 预览页面 HTML、CSS、JS
  - 已有 battlemap 根区域与轮询刷新逻辑
- `tools/services/encounter/get_encounter_state.py`
  - 负责把 Encounter 投影为 `encounter_state`

当前页面不是独立前端框架，而是服务端生成的 HTML 页面，因此本功能应沿用现有模式，不引入 React/Vue，也不额外拆前端构建链。

## 设计总览

### 页面结构

battlemap localhost 页面保留现有顶部 hero 和地图区域。

在地图区域下方新增一个独立的主角卡 section：

1. 左侧头像列
2. 右侧主面板列

其中右侧主面板再分为 3 层：

1. 顶部总览区
2. 六维属性区
3. 页签区与内容区

整体视觉方向：

- 颜色以暗铜、旧金、黑铁、焦褐为主
- 使用厚重边框、压暗底色、局部金属高光
- 避免纯表格 Excel 感
- 保持与现有 battlemap 页面深色氛围一致

## 详细布局

### 1. 主容器

新增 `player-sheet-shell` 容器，放在：

- `[data-role="battlemap-view-root"]` 后方

原因：

- 能明确形成“地图上 / 角色卡下”的阅读顺序
- 不影响现有地图 HTML 替换逻辑
- 后续扩展更多下方面板时结构清晰

### 2. 左侧头像区

固定大头像框，第一版支持：

- 静态占位图
- 无图时的默认头像占位样式

该区域第一版不做复杂交互，仅承担：

- 角色识别
- 视觉重心

### 3. 顶部总览区

显示：

- 角色名
- 职业 / 子职 / 等级
- `当前 HP / 最大 HP · AC`
- 法术豁免 DC
- 法术攻击加值

格式示例：

- `44 / 44 HP · AC 16`
- `法术豁免 13 · 法术攻击 +5`

### 4. 六维属性区

横向显示六项属性：

- 力量
- 敏捷
- 体魄
- 智力
- 感知
- 魅力

每项显示：

- 属性名
- 属性值
- 对应豁免值

第一版不显示额外派生说明，只保留最直接信息密度。

### 5. 页签区

底部使用按钮式页签，而不是直接把技能与装备长表摊开。

固定 3 个按钮：

- `技能`
- `装备`
- `后续追加`

要求：

- 同一时间只有一个激活态
- 激活态按钮有明显高亮
- 切换时仅更新内容区
- 不触发整页刷新

### 6. 内容区

#### 技能页

显示技能修正列表。

呈现方式：

- 2-3 列网格
- 每个技能一条简明记录
- 样式偏 RPG 面板，而不是原始大表格

字段包括但不限于：

- 运动
- 特技
- 巧手
- 隐匿
- 奥秘
- 历史
- 调查
- 自然
- 宗教
- 驯兽
- 洞悉
- 医疗
- 察觉
- 求生
- 欺瞒
- 威吓
- 表演
- 说服

#### 装备页

第一版重点显示武器条目，不尝试完整装备系统。

每个武器条目显示：

- 名称
- 命中
- 伤害
- 武器专精

若后续存在自定义字段，可在条目右侧预留扩展位，但第一版不要求后端完整支持。

#### 后续追加页

第一版为占位页签，用于后续扩展：

- 特性
- 状态
- 资源
- 法术位
- 被动效果

第一版仅显示占位文案与扩展说明，不塞临时假数据。

## 数据设计

前端不直接从零散 `encounter_state` 字段临时拼 UI，而是统一转换为一个 `player_sheet` 对象。

### 建议结构

```json
{
  "summary": {
    "name": "奎里昂",
    "class_name": "战士",
    "subclass_name": "奥法骑士",
    "level": 4,
    "hp_current": 44,
    "hp_max": 44,
    "ac": 16,
    "spell_save_dc": 13,
    "spell_attack_bonus": 5,
    "portrait_url": null
  },
  "abilities": [
    {"key": "str", "label": "力量", "score": 8, "save_bonus": 2},
    {"key": "dex", "label": "敏捷", "score": 18, "save_bonus": 4}
  ],
  "tabs": {
    "skills": [
      {"key": "athletics", "label": "运动", "modifier": -1}
    ],
    "equipment": [
      {
        "name": "干将",
        "attack_bonus": 7,
        "damage": "1d6+4",
        "mastery": "侵扰"
      }
    ],
    "extras": {
      "placeholder_title": "后续追加",
      "placeholder_body": "后续会加入特性、状态、资源与法术相关信息。"
    }
  }

```

### 第一版数据来源

第一版先在页面层定义静态 `player_sheet` 示例数据，用于确认 UI。

原因：

- 当前目标是尽快稳定出视觉与交互形态
- 避免一开始就被后端字段完整性拖慢
- 后续切动态时，不需要重写 UI 结构

### 第二版动态接入

第二版由页面增加一个适配层：

- `buildPlayerSheet(encounterState)`

职责：

- 从现有 `encounter_state` 中提取已有字段
- 组装成前端统一 `player_sheet`
- 缺字段时返回空值或占位值

最终收口方向：

- 后端在 `GetEncounterState` 中显式提供 `player_sheet` 或 `player_character_summary`
- 前端移除大部分临时映射逻辑，改为直接渲染后端投影

## 前端交互设计

### 页签状态

前端维护一个页面内状态：

- `activePlayerSheetTab`

默认值：

- `skills`

点击页签按钮时：

- 更新激活态
- 重绘内容区

### 状态刷新保持规则

当 polling 获得新的 `encounter_state` 后：

- 角色卡需要同步刷新
- 当前激活 tab **不得重置**

例如：

- 用户当前停在 `装备`
- 轮询刷新后仍然保持 `装备` 页签激活

### 容错

若某字段缺失：

- 不抛 JS 错误
- 用 `--`、空列表或占位文本兜底

例如：

- 没有头像：显示占位框
- 没有法术数据：顶部总览仍显示基础值，缺省项显示 `--`
- 装备为空：显示“暂无装备数据”

## 实现边界

### 不新增复杂前端架构

本功能继续沿用当前 localhost battlemap 模式：

- 服务端拼基础 HTML/CSS/JS
- 页面端用少量原生 JS 做状态切换和重绘

### 不与地图刷新机制耦死

地图区域当前通过：

- `window.applyEncounterState(nextState)`

替换 battlemap HTML。

角色卡不应依赖“地图 HTML 里顺手包含角色卡”，而应作为 battlemap 页面自身的一块独立 UI，由同一个 `nextState` 驱动刷新。

## 测试策略

### 需要覆盖的内容

1. localhost 页面初次渲染
   - 地图下方存在主角卡容器
   - 顶部总览存在 `HP · AC`
   - 默认激活 `技能`

2. 页签切换
   - `技能 / 装备 / 后续追加` 三按钮都可切换
   - 切换后内容区变化正确

3. 状态刷新
   - 调用 `window.applyEncounterState(nextState)` 后角色卡同步更新
   - 当前激活页签保持不变

4. 缺省值容错
   - 缺头像
   - 缺装备
   - 缺法术数据
   - 页面仍可渲染

### 建议测试落点

- `test/test_run_battlemap_localhost.py`
  - 验证 localhost 页面输出包含角色卡结构
- 如有需要，可补局部字符串断言：
  - 角色卡根节点
  - 页签按钮
  - `HP · AC` 摘要

## 实施顺序

### Phase 1: 静态 UI

- 在 `run_battlemap_localhost.py` 中加入角色卡区 HTML/CSS/JS
- 使用静态 `player_sheet` 示例数据渲染
- 打通 tab 切换

### Phase 2: 接现有状态

- 增加 `buildPlayerSheet(encounterState)` 前端适配层
- 用 `encounter_state` 驱动角色卡刷新
- 保留 tab 激活态

### Phase 3: 后端收口

- 在 `GetEncounterState` 中补正式角色卡投影字段
- 前端切换为优先使用后端 `player_sheet`

## 风险与对应处理

### 风险 1：当前 `encounter_state` 字段不够

处理：

- 第一版不阻塞，先静态
- 第二版允许临时适配层补空

### 风险 2：localhost 页面字符串拼接继续膨胀

处理：

- 本轮允许继续在 `run_battlemap_localhost.py` 落地
- 但新增角色卡相关 HTML/CSS/JS 时，应尽量集中成独立片段函数，而不是继续堆到一个超长字符串里

### 风险 3：后续扩展 tab 时返工

处理：

- 现在就固定 `tabs` 数据结构
- `后续追加` 提前预留

## 结论

第一版应以“尽快把地图下方主角卡做出来”为优先，先稳定：

- 视觉风格
- 页面布局
- tab 交互
- 前端统一数据对象

随后再逐步把静态数据替换成动态投影，避免在第一轮就把问题扩大成完整角色系统重构。
