# Web Battlemap Design

## 背景

当前 `trpg-dm-system` 已经有:

- `EncounterMap` 作为地图事实层
- `get_encounter_state` 作为战斗状态投影层
- 一份早期的 ASCII 地图设计草案

但继续沿着 ASCII 地图推进,已经暴露出几个明显问题:

1. `30 x 30` 级别的大地图对 LLM 和玩家都不够友好.
2. 想在地图中放职业 emoji 时,终端字符宽度和对齐不稳定.
3. ASCII 图既要承担"全局空间展示",又要承担"地形语义解释",职责过重.

因此,本次设计决定不再把 ASCII 视图作为主地图方案,而是改为网页化棋盘地图视图.

## 目标

第一阶段地图系统需要满足以下目标:

1. 玩家可以持续看到全局大地图.
2. 地图可以直观显示单位、墙、困难地形、高台、特殊区域.
3. 地图允许在格子里显示职业 emoji,不受终端字符对齐限制.
4. LLM 不只读地图图面,还能额外拿到结构化 `map_notes`,用于全局判断.
5. 底层规则仍然依赖结构化地图事实,不依赖渲染文本.

## 核心结论

### 1. 地图输出改为双层

地图相关输出分成两层:

- 玩家主视图:网页化全局 battlemap
- LLM 辅助视图:结构化 `map_notes`

这两层都来源于同一份 `EncounterMap` 事实层,不引入第二套地图真相源.

### 2. `EncounterMap` 继续作为事实层

`EncounterMap` 继续保存结构化地图数据,例如:

- `width`
- `height`
- `grid_size_feet`
- `terrain`
- `auras`
- `zones`
- 后续可扩展 `landmarks`

规则系统、查询服务、地图渲染都从这份事实层数据读取.

### 3. 网页 battlemap 取代 ASCII 成为主地图视图

主地图视图不再输出字符画,而是输出网页棋盘式网格:

- 每个格子对应一个真实 tile
- 每格固定可视尺寸
- 地形通过底色、边框、角标、标识符展示
- 单位通过 emoji 或图标展示
- 右侧保留 turn order / 当前行动者 / 图例 / 说明区

这样可以直接支持:

- 大地图常驻显示
- 玩家职业图标
- 更稳定的视觉层叠
- 后续扩展 hover、点击、局部高亮

### 4. `map_notes` 专门负责语义解释

`map_notes` 的职责不是重复地图图面,而是把关键地形和区域规则整理成 LLM 易读的结构化摘要.

它主要回答这些问题:

- 哪些区域是墙
- 哪些区域是困难地形
- 哪些区域是高台
- 哪些区域是危险区或法术区域
- 这些区域大概位于战场哪里
- 它们会造成什么规则影响

## 设计分层

### 1. Map Facts

事实层,来自 `EncounterMap`.

负责:

- 保存地图尺寸
- 保存地形格
- 保存区域格
- 保存光环和特殊区域
- 作为规则和渲染的统一来源

### 2. Battlemap View

玩家可见的网页地图视图.

负责:

- 全局网格地图
- 地形与区域视觉样式
- 单位图标显示
- 当前行动者高亮
- 图例和侧边信息区

### 3. Map Notes

提供给 LLM 的结构化地图摘要.

负责:

- 区域语义解释
- 规则影响摘要
- 关键地标和地形带说明
- 避免让 LLM 被大地图图面拖慢

## Battlemap View 设计

### 页面布局

建议采用左右布局:

- 左侧:主地图网格
- 右侧:战斗信息栏

右侧信息栏第一阶段建议包含:

- 当前回合信息
- turn order
- 图例
- 地图区域说明

### 单格渲染层级

每个格子按固定顺序渲染:

1. 底板
2. 地形层
3. 区域层
4. 单位层
5. 当前行动者高亮层

这样能保证后续扩展时优先级稳定,不会互相覆盖混乱.

### 地形与区域第一阶段范围

第一阶段先支持以下类型:

- 墙
- 困难地形
- 高台
- 特殊区域 / 法术区域

建议的视觉策略:

- 墙:深色或强边界格
- 困难地形:浅蓝或泥泞色底
- 高台:高亮地板 + 边框
- 特殊区域:发光或标记底纹

### 单位显示

友方单位允许按职业显示 emoji,例如:

- 野蛮人:`🪓`
- 吟游诗人:`🎼`
- 牧师:`⛪`
- 德鲁伊:`🐻`
- 战士:`🗡`
- 武僧:`🥋`
- 圣武士:`🛡`
- 游侠:`🏹`
- 盗贼:`🫥`
- 术士:`🧙‍♂️`
- 邪术师:`⚡`
- 法师:`📜`

敌方第一阶段可先使用统一图标或简化标识,后续再扩展怪物图标体系.

### 当前行动者

当前行动者不强制替换职业图标.

更稳的做法是:

- 保留职业图标
- 外加高亮边框、发光描边或状态环
- 在右侧信息栏明确标注当前行动者

这样不会损失单位识别信息.

## `map_notes` 设计

`map_notes` 应保持半结构化,而不是纯长篇自然语言.

建议包含以下块:

- `terrain_summary`
- `zone_summary`
- `landmarks`
- `tactical_warnings`

示例结构:

```json
{
  "terrain_summary": [
    {
      "type": "wall",
      "region": "east barrier",
      "cells": [[10, 4], [10, 5], [10, 6]],
      "rules": ["blocks_movement", "blocks_los"],
      "note": "东侧墙带把战场分成两段."
    }
  ],
  "zone_summary": [
    {
      "type": "difficult_terrain",
      "region": "central pool",
      "cells": [[5, 5], [5, 6], [6, 5], [6, 6]],
      "rules": ["costs_extra_movement"],
      "note": "中央水池区域移动成本加倍."
    }
  ],
  "landmarks": [
    {
      "type": "high_ground",
      "region": "northwest platform",
      "cells": [[2, 8], [2, 9], [3, 8], [3, 9]],
      "note": "西北角高台可俯视中央区域."
    }
  ]
}
```

设计原则:

- 先说区域是什么
- 再说覆盖哪些格子
- 再说规则影响
- 最后用一句自然语言帮助 LLM 建立全局理解

## 与现有服务的关系

### `get_encounter_state`

`get_encounter_state` 继续保留统一出口角色,但不在内部硬编码复杂渲染逻辑.

建议它在后续整合时返回:

- 原有战斗状态
- `battlemap_details`
- `battlemap_view`
- `map_notes`

其中:

- `battlemap_view` 由独立地图渲染 service 生成
- `map_notes` 由独立地图摘要 service 生成

### 建议新增服务

建议新增:

- `app/services/map/render_battlemap_view.py`
- `app/services/map/build_map_notes.py`

如果后续地图交互复杂,再补:

- `app/services/map/__init__.py`
- `app/services/map/view_models.py`

## 第一阶段最小实现

第一阶段只做最小可用闭环,不提前做过度扩展.

### 包含

1. 地图网页视图基础布局
2. 固定尺寸网格渲染
3. 支持墙、困难地形、高台、特殊区域
4. 支持友方职业 emoji
5. 支持敌方基础标记
6. 支持当前行动者高亮
7. 支持右侧 turn order / 图例 / 当前信息
8. 支持输出结构化 `map_notes`

### 暂不包含

1. 大体型单位占多格
2. 视野遮挡
3. 路径预览
4. AOE 模板交互高亮
5. 拖拽编辑
6. 动画和复杂过渡
7. 敌人专属图标体系

## 第一阶段成功标准

完成后应达到以下标准:

1. 玩家可以稳定看到一张全局大地图.
2. 地图能明显区分墙、困难地形、高台、特殊区域.
3. 友方职业可以直接在格子内显示.
4. 当前行动者能被明显识别.
5. LLM 能通过 `map_notes` 读懂战场全局语义,而不是只靠图面猜.
6. 底层地图事实仍然保持单一来源,没有把规则写死到渲染层里.

## 下一步

这份设计确认后,下一步应进入实现计划阶段,明确:

1. 地图网页视图放在哪个运行入口下
2. 是先做纯静态 HTML 渲染,还是直接接入现有系统输出
3. `EncounterMap` 是否需要补 `landmarks` 等字段
4. 第一批测试覆盖哪些服务和视图输出
