# ASCII 地图设计草案

## 目标

这份文档用于记录当前对 ASCII 地图模块的实现思路，方便后续继续开发。

当前目标不是做一个“好看的终端地图”，而是做一个：

- 对 LLM 易读
- 对规则系统可计算
- 对后续功能扩展友好

的战斗地图输出层。

## 核心判断

### 1. `EncounterMap` 继续只保存事实数据

`EncounterMap` 现在适合作为“地图事实层”，保存：

- `width`
- `height`
- `grid_size_feet`
- `terrain`
- `auras`
- `zones`

它不应该直接保存 ASCII 字符串。

原因：

- ASCII 地图是视图层产物，不是事实源
- 同一张地图未来可能有多种投影方式
- 如果把字符串直接存进运行态，后续维护会越来越乱

### 2. ASCII 地图应当由独立 service 渲染

建议新增一个地图服务目录：

```text
app/services/map/
├── __init__.py
└── render_ascii_map.py
```

职责划分：

- `Encounter` / `EncounterMap`
  - 保存地图和实体的事实数据
- `GetEncounterState`
  - 输出基础战斗视图
- `RenderAsciiMap`
  - 把地图事实数据渲染成 ASCII 地图

这样不会把“规则状态”和“地图显示”耦合在一起。

### 3. 不建议把 ASCII 渲染硬塞进 `get_encounter_state`

`get_encounter_state` 现在更像“通用战斗状态投影工具”。

如果把地图渲染细节直接塞进去，会有几个问题：

- 职责变重
- 测试会越来越难拆
- 后面加 viewport / 路径预览 / AOE 高亮时会很乱

更好的做法是：

- `get_encounter_state` 继续提供基础地图信息
- 地图细节由 `render_ascii_map` 单独提供

如果以后需要，也可以让 `get_encounter_state` 调用它，把结果挂到视图层里，但渲染逻辑本身不要放在 `get_encounter_state` 里面。

## 推荐的数据分层

### 1. 规则事实层

这是系统内部用于判定的地图数据。

建议继续结构化保存：

- `terrain`
  - 墙
  - 障碍
  - 困难地形
  - 危险地形
- `zones`
  - 法术区域
  - 光环区域
  - 特殊区域
- 以后可追加：
  - `traps`
  - `cover`
  - `hidden_objects`

这一层的目标是：

- 程序可自动判定
- 不依赖 LLM 自己读文本猜规则

### 2. 视图层

这是给 LLM 或 UI 看的内容。

建议输出：

- `ascii_map`
- `legend`
- `visible_entities`
- `viewport`
- `current_entity_id`

这一层的目标是：

- 让 LLM 能快速理解空间关系
- 不必直接消费底层全部结构化字段

## ASCII 地图的推荐返回结构

建议 `render_ascii_map.py` 最终返回：

```json
{
  "mode": "focus",
  "viewport": {
    "min_x": 11,
    "max_x": 19,
    "min_y": 15,
    "max_y": 23
  },
  "ascii_map": "......",
  "legend": {
    ".": "空地",
    "#": "墙体",
    "~": "危险地形",
    "*": "区域效果",
    "P": "玩家",
    "E": "敌人",
    "C": "当前行动者"
  },
  "visible_entities": [
    {
      "entity_id": "ent_pc_eric_001",
      "name": "Eric",
      "symbol": "C",
      "x": 15,
      "y": 19
    }
  ]
}
```

这里有几个关键点：

- `ascii_map` 给 LLM 直接阅读
- `legend` 减少 LLM 猜字符含义
- `visible_entities` 让 LLM 不必从整张图里反向找名字
- `viewport` 告诉 LLM 当前图只展示了哪一块区域

## 渲染时的分层顺序

建议渲染时按固定层级覆盖：

1. 底图层
   - 默认全部填 `.` 表示空地
2. 地形层
   - 例如 `#` 墙、`~` 危险地形、`^` 陷阱
3. 区域层
   - 例如法术区域、光环、燃烧区
4. 实体层
   - 玩家、敌人、召唤物、当前行动者

这样做的好处：

- 优先级清楚
- 测试容易写
- 后续增加新地图元素时不容易互相打架

## 字符方案建议

建议优先使用 ASCII 安全字符，不依赖中文字符宽度。

例如：

- `.` 空地
- `#` 墙
- `~` 危险地形
- `^` 陷阱
- `*` 区域效果
- `P` 玩家阵营
- `E` 敌对阵营
- `N` 中立单位
- `C` 当前行动者

说明：

- 如果当前行动者本身是玩家或敌人，`C` 的优先级高于阵营字符
- 图例里必须明确写清楚字符含义

不建议默认使用中文单字来画地图，因为终端宽度和对齐在不同环境中不稳定。

## 地图模式建议

### 1. `full` 模式

完整渲染整张地图。

适合：

- 调试
- 小地图
- GM 俯视查看全局

缺点：

- 30x30 以上时，对 LLM 阅读负担较大

### 2. `focus` 模式

以当前实体为中心，只渲染局部区域，例如 `9x9` 或 `11x11`。

适合：

- LLM 当前回合决策
- 玩家行动判断
- 快速看附近敌人、障碍、区域效果

这是后续更值得默认使用的模式。

## 第一阶段最小实现

建议先做最小可用版，不一次做太满。

### 第一阶段范围

- 新建 `app/services/map/render_ascii_map.py`
- 支持整图渲染
- 支持坐标轴
- 支持地形显示
- 支持实体显示
- 支持当前行动者标记
- 支持基础图例输出
- 写对应测试

### 第一阶段暂时不做

- 大体型单位占多格
- 视野遮挡
- 路径预览
- AOE 高亮
- 隐藏陷阱可见性控制
- `focus` viewport

## 第二阶段扩展

第一阶段稳定后，再做：

- `focus` 视窗
- `auras`
- `zones`
- 大体型单位占格
- 隐藏信息投影控制
- 与 `get_encounter_state` 的组合输出

## 与当前项目设计的一致性

这个方案和当前项目已经确定下来的设计是统一的：

- 视图层对象给 LLM 看
- 运行态快照作为事实源
- 事件日志负责记录变化
- service 负责计算和投影

所以地图模块也应当遵守同样原则：

- 地图事实数据存模型
- 地图 ASCII 投影由 service 计算
- 不把渲染结果反写为事实状态

## 我对商业方案的结论

之前讨论过一种“地图主要靠描述和 notes”的商业方案。

我的结论是：

- 如果目标是叙事型 AI DM，它是能用的
- 如果目标是本地可计算战斗系统，它不够扎实

因此当前项目更适合做成“两层地图”：

1. 规则事实层
2. 给 LLM 看的地图视图层

ASCII 地图就属于第二层。

## 下次继续开发时的建议顺序

1. 新建 `app/services/map/`
2. 实现 `render_ascii_map.py`
3. 为地图渲染补测试
4. 用 `data/examples/encounter_state.example.json` 做一次真实渲染验证
5. 再决定是否把 ASCII 地图挂进 `get_encounter_state`
